Не в каждом файле <processor> находится внутри <device>. Иногда он объявлен глобально внутри <family>

Некоторые устройства, такие как STM32 содержат внутри <family> ветку <subFamily>, в то время как Nordic такой ветки не содержит

В CoIDE принято такое разделение:
-- Manufacturer -> Family -> Serie -> Chip

Если провести аналогию с pdsc, то :
Manufacturer = <family Dvendor="STMicroelectronics:13"> (13 - это Vendor ID)
Family = <processor Dcore="Cortex-M4">
Serie = <subFamily DsubFamily="STM32F401"> или <family Dfamily="nRF51 Series"> (если нет subFamily)
Chip = <device Dname="nRF51801_xxAB">

То, что внесено в базу как Family, Serie и Chip средой автоматически генерируется в качестве Define, т.е. -DSTM32F401;

ID Manufacturer в базе CoIDE не соответствуют Vendor ID в системе Keil

Family в структуре CoIDE должны быть переименованы с "Cortex-M0" на "Cortex M0" (иногда автоматически вставляется дефайн -DM0;)

Алгоритмы программирование могут быть заданы как для семейства, так и для отдельного устройства. Например, Nordic DFP не содержит алгоритмов внутри device

значение default=1 может быть отмечено для нескольких алгоритмов (если они шьют разные регионы). Надо отслеживать

------------------------- Компоненты и взаимосвязь между ними и устройствами в базе данных ---------------------------------------
Каждый компонент имет:
- уникальный ID
- Layer ID (layer - это фактически вкладка на отображалке Repository в CoIDE):
    1	Driver      - отображается на вкладке Drivers
	2	MCU         - отображается на вкладке Peripherals (type = 2)
	3	User        - отображается на вкладке Others (бывает только с type = 2), применяется для mcu
	4	Middleware  - отображается на вкладке Peripherals (type = 2)
	5	Device      - отображается на вкладке Others (бывает только с type = 2), применяется для внешних устройств
	
- Type - 1 или 2, 1 - это всегда Drivers, 2 - это все остальное

В таблице components_has_category указан categoryId, к которой относится компонент.
В таблице components_has_subcategory указан subcategoryId, к которой относится компонент.

Таблица Category может содерждать произвольное название категории и может дополняться. От содержимого в Category.name зависит то,
что будет подписано в названии группы компонентов на вкладке Peripherals

Таблица Subcategory может содерждать произвольное название подкатегории и может дополняться. От содержимого в Subcategory.name зависит то,
что будет подписано в названии группы компонентов на вкладке Peripherals после точки.


Пример: Category.name = "peripherals", Subcategory.name = "ST" породит группу компонентов с названием "PERIPHERALS.ST" и будет видна на вкладке Peripherals

-------------
Читаем данные из таблицы component:
Component.id = 30, Component.name = "C Library", Component.Layer_id = 2, Component.type = 2

Читаем данные из таблицы components_has_category: componentId = 30, categoryId = 2
Читаем данные из таблицы category: Category.id = 2, Category.name = "Common"
В таблице components_has_subcategory нет данных о компоненте с id = 30

Отсюда следует, что компонент "C Library" будет отображаться в группе "COMMON" на вкладке "Peripherals"

ПРИМЕЧАНИЕ: Компонент обязательно должен быть записан в таблице component_supports_mcu. Его нельзя задать глобально
для какой-то серии в component_supports_mcuseries, семейства в component_supports_mcufamily или
производителя в component_supports_mcumanufacturer. Фактически записи дублируются, но не имеют силы.

ПРИМЕЧАНИЕ: Таблицы с префиксом component_not_supports_* не содержат данных и не несут никакой смысловой нагрузки
